# F-009 RAGAS 超时稳定性测试设计

## 功能流程测试
- 用例：助手消息保存后触发 RAGAS 评测。
- 输入：正常 question、answer、3 条以内中等长度 contexts。
- 预期行为：消息状态依次进入 `pending -> running -> done`，至少返回可用指标或部分指标。
- 潜在失败点：后台任务未启动、状态未落库、评测完成后前端仍保持 pending。

## 单指标超时隔离测试
- 用例：某一项 RAGAS metric 长时间无返回。
- 输入：模拟单 metric 执行超过 `RAGAS_METRIC_TIMEOUT_SECONDS`。
- 预期行为：超时 metric 写入 `_errors`，其余 metric 可继续完成；最终消息进入 `done` 而非整次 `failed`，前提是至少一项指标成功。
- 潜在失败点：单指标阻塞拖垮整次评测、线程未释放、错误文本不明确。

## 整体超时测试
- 用例：整次评测超过 `RAGAS_TIMEOUT_SECONDS`。
- 输入：模拟所有指标或上游网络整体阻塞。
- 预期行为：消息进入 `failed`，`ragas_error` 明确提示超时秒数，不应无限 pending。
- 潜在失败点：超时未生效、消息仍停留 running、错误提示缺失。

## 长上下文瘦身测试
- 用例：检索返回过长 contexts，answer 也较长。
- 输入：超过 `RAGAS_MAX_CONTEXTS`、单条 context 超过 `RAGAS_MAX_CONTEXT_CHARS`、answer 超过 `RAGAS_MAX_ANSWER_CHARS`。
- 预期行为：仅保留限定数量上下文，超长文本被截断后再送评测。
- 潜在失败点：仍原样传入导致 token 过大、截断后空文本处理异常。

## 边界与错误处理测试
- 用例：无 contexts、无 DeepSeek key、无 Embedding key。
- 输入：空 contexts 或缺失必要环境变量。
- 预期行为：能快速失败并写入中文错误，不出现卡死。
- 潜在失败点：异常冒泡导致任务未收口、错误信息不可读。

## 状态管理验证
- 用例：前端轮询遇到后端部分成功结果。
- 输入：`ragas_status=done` 且 `ragas_error` 含部分指标失败说明。
- 预期行为：前端停止轮询并展示已有分数，允许同时显示错误说明。
- 潜在失败点：前端误判为失败或继续轮询。
